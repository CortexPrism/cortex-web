# Plugin API Reference

Complete reference for the CortexPrism plugin system APIs: lifecycle hooks, capability interface, pipeline hooks, UI panel API, sandbox protocol, supply-chain verification, and manifest schema.

## Plugin Lifecycle Hooks

Plugins can hook into lifecycle events to perform initialization and cleanup:

```typescript
export const onInstall   = async (ctx: PluginContext) => {};
export const onLoad      = async (ctx: PluginContext) => {};
export const onActivate  = async (ctx: PluginContext) => {};
export const onDeactivate = async (ctx: PluginContext) => {};
export const onUnload    = async (ctx: PluginContext) => {};
export const onUninstall = async (ctx: PluginContext) => {};
export const onConfigChange = async (key: string, value: unknown, ctx: PluginContext) => {};
```

### Lifecycle Sequence

```
Install:     onInstall â†’ onLoad â†’ onActivate
Enable:      onLoad â†’ onActivate
Disable:     onDeactivate â†’ onUnload
Remove:      onDeactivate â†’ onUnload â†’ onUninstall
Reconfigure: onConfigChange (per key, non-blocking)
```

### Lifecycle Discipline

| Hook | What to do | What NOT to do |
|------|------------|----------------|
| `onInstall` | Write default state, log first install | Make network calls |
| `onLoad` | Subscribe to events, init in-memory state | Register tools |
| `onActivate` | Register tools, start polling | Slow blocking initialization |
| `onDeactivate` | Unregister tools, stop polling | Modify persisted state |
| `onUnload` | Unsubscribe events, close connections | Throw â€” breaks the unload sequence |
| `onUninstall` | Delete persisted state/config | Assume `onUnload` already ran |
| `onConfigChange` | Apply the new config value | Block â€” called per-key on every change |

## PluginContext API

Every lifecycle hook and tool `execute` receives a `PluginContext`:

| Property | Type | Description |
|----------|------|-------------|
| `pluginId` | `string` | Plugin name |
| `pluginDir` | `string` | `~/.cortex/data/plugins/<name>/` |
| `state` | `PluginStateStore` | Persistent SQLite-backed key-value store (survives reloads) |
| `config` | `PluginConfigStore` | Typed access to `~/.cortex/config.json plugins.<name>` |
| `logger` | `PluginLogger` | Scoped logger â€” prefixes `[plugin:<name>]` |
| `host` | `HostApi` | Register/unregister tools at runtime |

### PluginStateStore

```typescript
await ctx.state.set('counter', '0');
const counter = await ctx.state.get('counter');   // string | null
const all = await ctx.state.list();                // Record<string, string>
await ctx.state.delete('counter');
```

### PluginConfigStore

```typescript
const apiKey = await ctx.config.get<string>('apiKey');
await ctx.config.set('lastInit', Date.now());
```

### PluginLogger

Methods: `info`, `warn`, `error`, `debug`. Prefixes every message with `[plugin:<name>]`.

## Tool Implementation

### Tool Interface

```typescript
interface Tool {
  definition: {
    name: string;
    description: string;
    params: { name: string; type: string; description: string; required?: boolean; enum?: string[] }[];
    capabilities: string[];  // per-tool permissions, subset of manifest capabilities
  };
  execute(args: Record<string, unknown>, ctx: ToolContext): Promise<ToolCallResult>;
}

interface ToolCallResult {
  toolName: string;
  success: boolean;
  output: string;           // always a string â€” JSON.stringify objects
  error?: string;
  durationMs: number;
}
```

### Tool Best Practices

- Validate inputs first
- Never throw from `execute()` â€” it breaks the tool loop. Always return a `ToolCallResult`.
- Always return a string for `output` â€” use `JSON.stringify()` for structured data.
- Match tool capabilities to manifest: tool capabilities must be a subset of manifest capabilities.

```typescript
// GOOD â€” always return a result
execute: async (args) => {
  const start = Date.now();
  try {
    const result = await doWork(args);
    return { toolName: 'my_tool', success: true, output: result, durationMs: Date.now() - start };
  } catch (err) {
    return { toolName: 'my_tool', success: false, output: '', error: (err as Error).message, durationMs: Date.now() - start };
  }
},
```

## Pipeline Hooks

Plugins can intercept the agent's 12-stage execution pipeline. Export `middlewarePre` and/or `middlewarePost` from your module and declare `middleware:pre` / `middleware:post` in capabilities.

### Pipeline Stages

```
INPUT â†’ pre-assess â†’ ASSESS (metacognition) â†’ post-assess
  â†’ pre-reason â†’ REASON â†’ post-reason
  â†’ pre-llm â†’ LLM CALL â†’ post-llm
  â†’ pre-tool â†’ TOOL EXECUTE â†’ post-tool
  â†’ pre-reflect â†’ REFLECT â†’ post-reflect
  â†’ pre-output â†’ OUTPUT â†’ post-output
```

| Stage | When | Available context fields |
|-------|------|--------------------------|
| `pre-assess` | Before metacognitive assessment | `input`, `messages` |
| `post-assess` | After assessment | `assessment`, `messages` |
| `pre-reason` | Before reasoning loop | `input`, `messages` |
| `post-reason` | After reasoning loop | `output` |
| `pre-llm` | Before LLM API call | `messages` |
| `post-llm` | After LLM API call | `currentLLMResponse` |
| `pre-tool` | Before each tool call | `toolCall` |
| `post-tool` | After each tool call | `toolCall`, `toolResult` |
| `pre-reflect` | Before reflection | `output` |
| `post-reflect` | After reflection | `reflection` |
| `pre-output` | Before final response | `output` |
| `post-output` | After final response | `output` |

### HookResult Fields

```typescript
interface HookResult {
  abort?: { reason: string; message: string };           // abort the turn
  modifyInput?: string;                                   // replace user input
  modifyLLMResponse?: string;                            // replace raw LLM response text
  modifyOutput?: string;                                 // replace final output
  injectMessages?: Message[];                            // append to conversation history
  sideEffects?: Array<{ type: 'log'|'metric'|'store'|'notify'; payload: unknown }>;
}
```

### Hook Best Practices

```typescript
// Always guard on stage
export const middlewarePre = async (ctx: unknown) => {
  const c = ctx as { stage: string; toolCall?: { name: string } };
  if (c.stage !== 'pre-tool') return {};
  if (!c.toolCall) return {};
  // your logic here
  return {};
};

// Provide clear abort messages
return { abort: { reason: 'policy', message: 'File access is blocked outside ~/projects/.' } };

// Keep hooks fast â€” offload slow work
export const middlewarePost = async (ctx: unknown) => {
  const c = ctx as { stage: string; toolResult?: { toolName: string } };
  if (c.stage !== 'post-tool' || !c.toolResult) return {};
  sendAuditLog(c.toolResult).catch(() => {});
  return {};
};

// Use sideEffects for structured logging
return {
  sideEffects: [
    { type: 'store', payload: { key: 'lastTool', value: c.toolCall?.name } },
    { type: 'log', payload: { tool: c.toolCall?.name } },
  ],
};
```

Sync hooks time out after **5s**. Async hooks time out after **15s**. Plugin middleware hooks run at `priority: 50`.

## Event Subscriptions

Declare `events:listener` in capabilities and list event types in the manifest `events` field. Subscribe in `onLoad`, unsubscribe in `onUnload`.

| Event | Payload |
|-------|---------|
| `session:start` | `{ sessionId }` |
| `session:end` | `{ sessionId }` |
| `tool:pre-execute` | `{ toolName, args }` |
| `tool:post-execute` | `{ toolName, result }` |
| `llm:pre-call` | `{ provider, model }` |
| `llm:post-call` | `{ provider, model, tokensIn, tokensOut }` |
| `agent:turn-start` | `{ sessionId, turnId }` |
| `agent:turn-end` | `{ sessionId, turnId, response }` |
| `config:change` | `{ key, value }` |
| `daemon:status` | `{ daemon, status }` |

## Sandbox Protocol (JSON-RPC over postMessage)

`untrusted` and `signed` plugins run in a **Deno Worker sandbox** using JSON-RPC over `postMessage`:

1. Host spawns `new Worker(entryPoint, { type: 'module', deno: { permissions: ... } })`
2. Worker must post `{ type: 'ready' }` within **30 seconds**
3. Host sends RPC: `{ id, method: 'getTools', params: {} }` â†’ worker returns tool definitions
4. On tool call: `{ id, method: 'executeTool', params: { toolName, args } }` â†’ worker returns `ToolCallResult`

### Sandbox Safety Rules

If your plugin may run sandboxed:
- Do not use `Deno.env` â€” env access is restricted in the Worker
- Do not use `Deno.Command` â€” requires `shell:run` capability
- Do not call `Deno.exit()` â€” terminates the Worker
- Do not import from `https://` at runtime without `network:fetch`
- Test with minimal permissions

## `window.Cortex` Panel API

Injected into every plugin panel iframe:

| Method | Description |
|--------|-------------|
| `window.Cortex.fetch(path, init?)` | Authenticated fetch to CortexPrism API |
| `window.Cortex.getConfig(key)` | Read plugin config value |
| `window.Cortex.setConfig(key, value)` | Write plugin config value |
| `window.Cortex.onEvent(event, handler)` | Subscribe to event bus |
| `window.Cortex.emit(event, data)` | Emit event to host |
| `window.Cortex.notify(msg, type)` | Show notification (type: 'info' \| 'warn' \| 'error') |

### PostMessage Commands

Structured commands panels can send to the host: `navigate`, `open-modal`, `notification`, `config-get`, `config-set`, `query`.

### UI Panel Best Practices

- Sanitize dynamic content â€” never render raw API data as HTML
- Use `window.Cortex.getConfig()` instead of hardcoded values
- Use `window.Cortex.notify()` for user feedback â€” avoid `alert()`
- Handle fetch failures â€” `window.Cortex.fetch()` can fail
- Design for narrow widths â€” sidebar panels may be as narrow as 240px
- Never trust panel input â€” validate data from panels server-side

## Trust Levels

| Level | Sandbox | How Assigned |
|-------|---------|--------------|
| `untrusted` | Deno Worker sandbox (minimal perms) | No verification, or blocked/suspicious supply-chain result |
| `signed` | Deno Worker sandbox | Supply-chain ran but no hash match (unverified) |
| `trusted` | In-process (full declared perms) | Supply-chain verification passes |

Trust level is set at install time and can be changed via `cortex plugins permissions <name>`.

## Supply-Chain Verification

`verifyPluginIntegrity()` runs before every plugin install:

- **SHA-256 hash** check against known-good hashes per `package@version`
- **Blocked hash list** â€” known-malicious hashes are rejected outright
- **Digital signature verification** â€” optional GPG signature validation
- **Author reputation scoring** â€” 0â€“100 score; configurable blocked/allowed author lists
- **Malware pattern scanning** â€” 6 default patterns

`blockSuspicious` mode rejects suspicious plugins. `requireKnownHash` defaults to `false` for community compatibility.

## Permission Overrides

Admins can grant or deny capabilities per-plugin:

```
POST /api/plugins/:name/permissions
{ "permission_path": "shell:run", "action": "deny" }
```

Effective permissions: `declared âˆ’ denied + granted`

## Plugin Namespacing

Marketplace plugins use `@author/name` scoping. Tool names from namespaced plugins follow `@author/plugin-name/tool_name`.

## MCP Plugin Development

MCP plugins wrap an external JSON-RPC 2.0 server. `entryPoint` must be an `https://` URL. CortexPrism creates a synthetic tool `mcp_<name>` that proxies calls.

## WASM Plugin Development

WASM plugins compile to `.wasm` and run in a WebAssembly sandbox. Required exports: `plugin_init`, `plugin_get_capabilities`, `plugin_execute_tool`, `memory`. Optional: `plugin_destroy`. Host import functions provided as `env.*`.

Memory layout: `initial: 256 pages` (16 MiB), `maximum: 512 pages` (32 MiB).

## Registry DB Schema (plugins.db)

The plugin registry is stored in `plugins.db` with this schema:

```sql
CREATE TABLE plugins (
  id          TEXT PRIMARY KEY,
  name        TEXT NOT NULL UNIQUE,
  version     TEXT NOT NULL,
  type        TEXT NOT NULL CHECK(type IN ('esm','mcp','wasm')),
  entry_point TEXT NOT NULL,
  manifest    TEXT NOT NULL,       -- Full manifest as JSON
  enabled     INTEGER DEFAULT 1,
  installed_at TEXT NOT NULL,
  updated_at  TEXT NOT NULL,
  download_count INTEGER DEFAULT 0
);

CREATE TABLE plugin_permissions (
  id          TEXT PRIMARY KEY,
  plugin_id   TEXT NOT NULL REFERENCES plugins(id),
  permission  TEXT NOT NULL,
  granted     INTEGER DEFAULT 0
);

CREATE TABLE plugin_config (
  plugin_id TEXT PRIMARY KEY REFERENCES plugins(id),
  config    TEXT NOT NULL DEFAULT '{}'
);
```

## Debugging

| Symptom | Cause / Fix |
|---------|-------------|
| Status: `error` | Check `error_message` via `cortex plugins list` |
| Tool not visible | Plugin must be `active`; tool must be in `tools` export |
| Permission denied | Tool `capabilities` must be subset of manifest `capabilities` |
| UI panel missing | Declare `ui:panel`; verify `htmlPath` is relative |
| Events not firing | Declare `events:listener` AND list event types in manifest |
| Sandbox not ready | Worker must post `{ type: 'ready' }` within 30s |
| Config not reading | Key must match what was set via `ctx.config.set()` |

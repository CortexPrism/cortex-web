# Pipeline System

The pipeline system provides a hook-based middleware architecture that wraps every stage of the agent turn lifecycle. Core and plugin hooks execute at 12 distinct pipeline stages with configurable priority, timeout protection, and rich modification capabilities.

## Architecture

```
┌──────────────────────────────────────────────────────────┐
│                    Pipeline System                         │
│                                                           │
│  12 Pipeline Stages (in order):                            │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐               │
│  │pre-assess│→│post-assess│→│pre-reason│→ ...             │
│  └──────────┘  └──────────┘  └──────────┘               │
│       │              │              │                      │
│       ▼              ▼              ▼                      │
│  ┌────────────────────────────────────────────┐          │
│  │          Registered Hooks (prioritized)     │          │
│  │  Core Hooks + Plugin Hooks                  │          │
│  └────────────────────────────────────────────┘          │
└──────────────────────────────────────────────────────────┘
```

## 12 Pipeline Stages

```
INPUT → pre-assess → ASSESS (metacognition) → post-assess
  → pre-reason → REASON → post-reason
  → pre-llm → LLM CALL → post-llm
  → pre-tool → TOOL EXECUTE → post-tool
  → pre-reflect → REFLECT → post-reflect
  → pre-output → OUTPUT → post-output
```

| Stage | When | Purpose |
|-------|------|---------|
| `pre-assess` | Before metacognitive assessment | Pre-filter, context preparation |
| `post-assess` | After assessment | Validate assessment output |
| `pre-reason` | Before reasoning loop | Modify reasoning context, context compaction |
| `post-reason` | After reasoning loop | Process reasoning output, completion checklists |
| `pre-llm` | Before LLM API call | Model selection, request modification |
| `post-llm` | After LLM API call | Response processing, model metrics |
| `pre-tool` | Before tool execution | Tool call filtering, loop detection |
| `post-tool` | After tool execution | Tool result processing, DLP scanning, cost tracking |
| `pre-reflect` | Before reflection | Modify reflection context |
| `post-reflect` | After reflection | Process reflection output |
| `pre-output` | Before final output | Content safety, DLP guard, modify response |
| `post-output` | After final output | Audit logging, cost tracking, bias auditing |

## Hook Interface

```typescript
interface PipelineHook {
  name: string;
  stages: PipelineStage[];
  priority: number;
  async: boolean;
  disableable: boolean;
  run(ctx: PipelineContext): Promise<HookResult>;
}

interface HookResult {
  abort?: boolean;
  modifyInput?: string;
  modifyLLMResponse?: string;
  modifyOutput?: string;
  injectMessages?: Message[];
  sideEffects?: SideEffect[];
}
```

## Hook Actions

| Action | Description |
|--------|-------------|
| `abort` | Stop the pipeline with a message |
| `modifyInput` | Mutate the input before processing |
| `modifyLLMResponse` | Modify the LLM's raw response text |
| `modifyOutput` | Modify the final output |
| `injectMessages` | Add messages to the conversation |
| `sideEffects` | Log, emit metrics, store data, send notifications |

## Execution Model

- **Sync hooks** — block the pipeline; 5-second timeout per hook
- **Async hooks** — fire-and-forget; 15-second timeout per hook
- **Priority-ordered** — lower priority numbers execute first within each stage
- **Abort support** — any hook can return `{ abort }` to stop the pipeline immediately
- Plugin middleware hooks run at `priority: 50`

## Hook Registration

Hooks can be registered by:
- Core system on startup
- Plugins via the plugin manager (export `middlewarePre` / `middlewarePost`)
- Custom scripts via the hooks API

Each hook has a priority determining execution order within a stage.

## Pipeline Context

Hooks receive a read-only `PipelineContext` with:
- `sessionId`, `turnId` — Current session/turn identifiers
- `tokensUsed`, `costUsd` — Current consumption metrics
- `toolCallsMade` — Count of tool calls this turn
- `setState(key, value)` — Mutate pipeline state
